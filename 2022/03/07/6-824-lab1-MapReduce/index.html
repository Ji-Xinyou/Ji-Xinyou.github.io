¶general picture

The MapReduce in general has two sides, one master and several workers.

The master and worker use messages to communicate

  • The master uses messages to distribute tasks (map or reduce) to workers
  • The worker uses messages to tell the master that it needs a task or its task is done

¶

In real world, MapReduce works on Distributed File System like GFS, in which large files are split into small chunks of 64MB. In the lab the files are around the order of some KB, so it is fine to MAKE EACH FILE A MAP TASK

The # of Reduce task is set by user, actually in function makeMaster()

Therefore, we have the result that

  1. we have file # number of map task, each corresponds to a file
  2. we have nReduce number of reduce task, each corresponds to a output file

Let’s have a look at the stages of MapReduce job, the figure is from the paper. This execution model is very important for both comprehending and implementing.

  • Split: for this lab, we split the input files a chunk per file - that is no split
  • Read: map worker read the file locally in the lab, but actually a mixture of RPC and local read in the real world
  • Intermediate:
    • It is important to figure out how many intermediate files does a MapReduce job produce.
    • According to the hint, we name the intermediate file mr-x-y, where x is map task #, y is reduce task #, this representation is easier to understand.
    • from above, we know we have file # of x, and nReduce of y
    • THEREFORE, we know, for each Map task # x, if generates nReduce intermediate file
      • mr-x-0, mr-x-1, mr-x-2, …, mr-x-(nReduce-1)
    • IMPORTANTLY, the job phase moves FROM Map phase TO Reduce phase, which means only when ALL MAP TASKS ARE DONE, can any Reduce task be distributed to workers
  • Reduce read: decode from json file (act as a substitution of remote read)

¶master (coordinator)

The master works as a state machine.

This is because the master is the one who detects the state and change the state of the whole job, which means the master must remember necessary states of the job for changes

The changes includes

  • Phase changing
    • from Map -> Reduce -> Done
    • No need to holds a UNSTARTED phase, starting from Phase_Map is fine
  • Crash recovery
    • When some worker takes the task, the master changes per-task phase. But what if the worker exits for some unknown reasons.

The states Master struct holds could look like this.

It changes the jobphase by doing checks on its state variables.

  • If you want to use unsafe data structures like hashmap, you need at least a lock to make some operations on them atomic
